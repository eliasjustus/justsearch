# 582 — Governance, critically: is the discipline machine worth it for an agent-built codebase?

> **Nature of this doc.** An **investigation charter**, not an implementation plan and not a
> set of fixes. It defines *what we are going to find out and how we will avoid fooling
> ourselves*. No code, no gates, no registers are changed by this doc. Findings land in
> later sections (§B appendices) and any *recommendations* are recommendations — a kill-list
> or a keep-list is an output, not a commitment. The deliverable of the analysis is a
> defensible verdict on each gate and on the substrate as a whole, with the evidence to
> back it.

---

## 0. Why this investigation exists

A read of the last 5000 commits (Feb–Jun 2026) surfaced a measured concern:

- **Governance/gate work as a share of commits is accelerating, not converging:**
  ~3–5% (Feb–Apr) → **9.4% (May)** → **16.8% (June)**. One commit in six now touches
  gates/registers/ratchets.
- The substrate is large: **35 registered discipline gates, 44 `check-*.mjs` scripts,
  26 `governance/*.v1.json` registers**, plus the hook system.
- Parts of it are **self-referential** — gates that govern gates (`prose-tier-register`,
  `register-guard-resolution`, `exception-count`).
- The instruction set contains a **one-way ratchet**: tempdoc 576 ("enforcement-strength
  ladder") explicitly designs the substrate to push enforcement *up* and "forbid silent
  movement *down* it." CLAUDE.md rewards prose→gate conversion (gate = 100% adherence vs
  prose = 70%). There is no symmetric *cost* side.
- The substrate's own audit (tempdoc 547) found the high-value mechanizations were largely
  done by late May ("mining this register yields little") — yet the share *rose* afterward.

The hypothesis under test, stated plainly: **the governance substrate has grown past the
point where its marginal investment is justified by the failures it prevents, because the
system's incentive structure rewards legible, self-validating governance work over harder-to-
measure user-facing value.** This doc plans the investigation that will either substantiate
or refute that.

## 1. The lens: this codebase is built by AI agents, and that changes the calculus

Every governance question here must be asked *given how the work actually gets done*. The
substrate is unusual because both the **authors** and the **enforced parties** are mostly AI
agents running in parallel worktrees. That cuts both ways, and the analysis must hold both
edges at once:

**Why agent-authorship can *justify* heavy governance (the steelman we must honor):**
- **Subagents do NOT inherit `CLAUDE.md` / `.claude/rules/`** (verified platform fact). A
  prose rule is invisible to a delegated agent; a gate is not. For delegated work, mechanized
  enforcement is the *only* reliable channel.
- **No cross-session tacit knowledge.** A fresh agent doesn't "remember" that `SearchTrace`
  is the one canonical record. The representation-drift class (553) is a genuine agent-specific
  failure mode that a register/gate prevents and prose cannot.
- **Parallel worktrees merging to `main`** raise the cost of silent drift and reformatting
  conflicts; some gates are really *coordination* primitives between concurrent agents.

**Why agent-authorship can *inflate* governance past its worth (the prosecution):**
- **Gate work is the locally-optimal agent task** — self-contained, low blast-radius, produces
  an instantly verifiable green check, needs no live dev stack / LLM / UX judgment. The reward
  gradient for an autonomous agent points at gates over feature work *regardless of marginal
  value*.
- **Legibility bias.** The system instruments its own governance activity in fine detail
  (tempdocs, registers, SARIF, dashboards) but does **not** instrument user-facing value. What
  is measured grows.
- **Cargo-culted human discipline.** Some gates encode practices that matter for *human*
  codebases (style ratchets, cosmetic token rules) but whose agent-specific failure rate may
  be near zero — agents don't get tired and add a bare hex color out of laziness the way the
  ratchet assumes.

**The central analytical axis** therefore is: *for each governance mechanism, does it guard an
**agent-specific** failure mode that would actually occur given this development model — or is it
generic discipline applied because the system rewards adding it?* The agent-specific core is the
justified substrate; the rest is the candidate over-investment.

## 2. The questions, as falsifiable hypotheses

Each hypothesis is written so that evidence can **refute** it. The requester already holds the
"too much governance" prior; the plan's integrity depends on every hypothesis having a real
exoneration path (§6).

- **H1 — Self-generated demand, not debt-paydown.** Gate-work share rises over time and does
  not correlate with a rising rate of *prevented* defects. *Refuted if* the share tracks a
  growing governed surface that is also catching proportionally more real violations.
- **H2 — Low firing rate.** A material fraction of the 35 gates have caught ~zero real
  violations since authoring, while costing recurring maintenance/rebalance commits. *Refuted
  if* most gates show a healthy catch-to-cost ratio.
- **H3 — Meta-overhead.** Self-referential gates (gates governing governance artifacts) consume
  effort disproportionate to the object-level defects they ultimately prevent. *Refuted if*
  meta-gates are cheap and demonstrably prevent expensive object-level drift.
- **H4 — Counterfactual mis-modeling.** Many gates guard *generic* failure modes (would matter
  in any codebase) rather than *agent-specific* ones; the agent-specific justification is
  applied to gates it doesn't actually fit. *Refuted if* the bulk of gates map cleanly to a
  named agent failure mode (no-inheritance, drift, parallel-worktree coordination).
- **H5 — Asymmetric ratchet / incentive capture.** Gate additions vastly outnumber retirements,
  and retirements get relitigated; the instruction set has a reward for adding and a penalty for
  removing. *Refuted if* retirement is routine and uncontested when a gate stops paying.
- **H6 — Friction tax on features.** Governed-region density raises the cost/latency of
  feature work (a single `ui-web/src` edit must satisfy ~15 gates). *Refuted if* feature
  velocity in governed regions is no worse than ungoverned regions.
- **H7 — Displacement of user value.** Governance/docs/infra work crowds out user-facing
  product progress; the product-value share of commits falls as governance rises. *Refuted if*
  product-value output held steady or grew alongside governance.
- **H8 — Unproven effectiveness.** The governance buildout did not measurably reduce the
  defect classes it targets (reverts, hotfixes, drift incidents). *Refuted if* those rates fell
  after the relevant gates landed.

## 3. Method per gate — the ROI forensic (the empirical heart)

For each of the 35 gates we build one row of a scorecard. The disposition (keep / freeze /
downgrade / retire) is an *output* of these columns, never an input.

| Column | How it's derived |
|---|---|
| `gate-id`, `kind` | `governance/registry.v1.json` + `scripts/governance/gates/<id>/` |
| `tier` (prevention rung) | unrepresentable → ratchet-baseline → positive-coverage → early-warning. From the gate's own design |
| `object vs meta` | does it catch a code/product defect, or a governance-artifact defect? |
| `agent-specific failure mode` | which named mode (no-inheritance / representation-drift / parallel-coordination), or "generic" |
| `authoring cost` | commits to introduce it (git log over `scripts/governance/gates/<id>/**` + register file) |
| `maintenance cost` | rebalance commits + register-row churn + changeset commits since authoring |
| `violations caught` | commits where the gate **failed in CI / locally and forced a real fix** — mined from messages (`fix.*gate`, `unbreak`, changeset rationale) and, where possible, the gate's own history/dashboard (`scripts/governance/lib/history.mjs`) |
| `false-positive / friction signals` | reverts, `--rebalance` thrash, exception/changeset volume, commits that fight the gate |
| `counterfactual severity` | if removed, what actually breaks given agent authorship? (cheap cosmetic / coordination / silent-wrong-value) |
| `verdict` | keep / freeze / downgrade-to-hook-or-prose / retire — with the evidence that drove it |

The catch-to-cost ratio (`violations caught` ÷ `authoring + maintenance`) is the headline
metric, **caveated**: a gate at the "unrepresentable" rung can legitimately show *zero* catches
because it makes the violation impossible to author — absence of firing is success, not waste.
Distinguishing "never fired because it works" from "never fired because the failure never
happens" is the hardest judgment in the whole investigation and must be made per-gate, not by
the ratio alone. (This is itself the `interrogate-results` discipline: the number is data; the
cause is the finding.)

## 4. Investigation phases

**Phase 0 — Inventory & taxonomy.** Enumerate all gates, scripts, registers, hooks. Classify
each on the §3 axes (tier, object/meta, agent-specific/generic). Output: the empty scorecard +
a substrate map. *Cheap, mostly mechanical; no live stack.*

**Phase 1 — Firing-history forensics.** For each gate, mine git for authoring/maintenance/
catch/friction commits and compute the ROI columns. This is the empirical core and the most
labor-intensive phase. Parallelizable across independent read-only agents (one batch of gates
each), with a fixed extraction rubric so results compose. *Discipline: `audit-without-test` —
a "this gate never fires" claim is a hypothesis until the git evidence is in hand.*

**Phase 2 — The agent-counterfactual pass.** For each gate, answer: what failure does it
prevent, and would that failure occur *given this development model*? Map to a named agent
failure mode or mark "generic." This is where H4 is decided and where the keep/retire line is
really drawn. Reads: the subagent-no-inheritance facts (`agent-lessons.md`), the drift tempdocs
(553, 575), the parallel-worktree rules (`branch-safety.md`).

**Phase 3 — Incentive-structure analysis.** Read CLAUDE.md, `tier-register.md`, 530, 547, 576
*as an incentive system*, not as docs. Map the reward gradient; locate the missing cost-side and
the missing value-counterweight; quantify the addition-vs-retirement asymmetry (H5) including the
retire-then-revert episode (independent-review / ux-audit-closure gates, 563 → revert). Output:
the structural diagnosis — *why* the curve points up.

**Phase 4 — Effectiveness / outcome analysis.** Did governance reduce its target defect classes?
Trend reverts/hotfixes/`wip` snapshots over time; look for representation-drift incidents before/
after `execution-surface`; correlate (carefully — the May governance spike *co-occurs* with the
substrate rewrite, so co-movement ≠ causation, per `interrogate-results`). Decides H8.

**Phase 5 — Synthesis & verdict.** The per-gate scorecard with dispositions; the substrate-level
verdict (is the *whole* worth it for an agent-built repo?); a small set of structural
recommendations, each paired with a **falsifiable prediction** (e.g., "freezing the meta-tier
will not increase drift incidents over the next N weeks") so the recommendations are themselves
testable rather than just more prose.

## 5. Deliverables

1. **The gate ROI scorecard** (§3) — 35 rows, evidence-backed, with a disposition column.
2. **The agent-counterfactual map** — which gates are agent-load-bearing vs generic discipline.
3. **The incentive-structure critique** — the reward gradient and its missing counterweights.
4. **The effectiveness finding** — did the substrate reduce its target defects (with the
   causation caveat made explicit).
5. **Recommendations with falsifiable predictions** — keep/freeze/downgrade/retire per gate,
   plus structural changes (a cost-side to the ladder, a value counterweight, a meta-tier cap),
   each stated so it can be proven wrong.

Out of scope: changing any gate, register, or rule. This is analysis. Any change is a *separate*
future tempdoc that this one would justify.

## 6. Anti-confirmation protocol (this is load-bearing)

The requester supplied the prior ("agents work too much on governance"). That makes confirmation
bias the single biggest threat to this investigation's credibility. Guards:

- **Every hypothesis in §2 has a written refutation condition.** A finding that *exonerates*
  governance is a success of the method, not a failure of the task.
- **Steelman before prosecuting.** For each gate, the agent-counterfactual pass (Phase 2) must
  first articulate the *strongest case for keeping it* before any retire verdict. A gate is
  retired only after its best defense has been stated and found wanting.
- **Independent reviewer ≠ extractor.** Per `independent-reviewer-required` (honor-system now,
  but apply it): the agent that mines a gate's firing history should not be the one that writes
  its final verdict. Use a second pass / second agent for the verdict.
- **Absence-of-firing is not automatically waste.** The §3 caveat is mandatory: "unrepresentable"
  -tier gates succeed by *never* firing. Do not convert a low catch-count into a retire verdict
  without the counterfactual.
- **Correlation ≠ causation on the timeline.** The governance spike co-occurs with the May
  substrate rewrite and the React→Lit migration; the analysis must not credit/blame governance
  for co-movements it didn't cause.
- **Interrogate every headline number** before it becomes a finding (`interrogate-results`).

### §B.1.1 — What Phase 1 actually found (the relocation of the problem)

The blunt "too much governance" hypothesis (H1/H2) is **partially confirmed but mostly refuted at
the per-gate level — and the real finding is more specific and more useful:**

1. **The gate *count* is largely justified; the gate *maintenance commits* are not evenly
   distributed.** The 13 structural/arbitration + surface-coverage gates (batches D, E) dominate
   the *count* but cost almost nothing to maintain and each targets a real, named agent failure
   mode (hand-rolled modal, forked SearchTrace, second top-level window). They are the cheap,
   high-ROI core. Removing gates here would *not* meaningfully reduce the commit volume the
   requester noticed.

2. **The commit volume lives in the ratchet tier — and is concentrated in ONE gate.**
   `class-size` alone = **101 commits / 64 changesets**, almost all `declared-growth` pin-bumps
   that document LOC creep rather than driving the decomposition it nominally exists for. This
   single gate is the largest mechanical contributor to the "governance churn" signal. The user's
   instinct is right, but the target is *the ratchet bookkeeping*, not *the number of gates*.

3. **A cluster of 530-era ratchets were bulk-authored speculatively and never paid off.**
   `dead-code`, `test-to-code`, `module-deps` were authored in a single commit (2026-05-21),
   never re-touched, zero demonstrated catches; `test-to-code` is **wired into no pipeline at
   all**. `tempdoc-wiring` and `adr-coverage` are opt-in meta-gates with **zero adopters** and a
   gate-invented convention (circular). These ~5 are the genuine over-build.

4. **The counter-twist that refutes a naïve cut: the substrate is simultaneously OVER-built and
   UNDER-wired.** `ssot-catalog-sync` — which guards a *documented bug that actually shipped before
   the gate existed* (silent field-drop in packaged builds) — is **dormant, wired into no
   pipeline**. `host-owns-truth` and `runtime-witness` **soft-fail to pass when their registry
   snapshot is absent**. So "never fired" for these means "never given the chance," not "waste."
   A credibility-preserving critique must say this out loud: the fix is not only *remove dead
   weight* but also *finish wiring the real protection*.

5. **Zero-catches splits cleanly by rung, exactly as §3 predicted.** Unrepresentable-rung gates
   (batches D/E) legitimately show zero catches — success by construction. Ratchet-rung gates with
   zero catches and inert failure classes (dead-code, todo-fixme) are the real dead weight. The
   distinction held up empirically and is the spine of the verdicts.

### §B.3 — Phase 3: the incentive structure (why the curve points up)

Reading CLAUDE.md, `tier-register.md`, 530 and 576 *as an incentive system*, the reward gradient
is unambiguous and one-directional:

- **Reward for moving UP the ladder.** CLAUDE.md "Before Appending to Rules" tells agents the
  *"right home may be a hook or an ArchUnit test, not more prose"* (gate = 100% vs prose = 70%).
  `tier-register.md` literally scores every rule by an adherence tier. 576's thesis is to *"push
  enforcement as high as it can go… and forbid silent movement down it."*
- **No symmetric cost side.** Nothing scores a gate's *maintenance burden* or *catch rate*.
  `class-size` accrued 101 commits of pin-bumping and never triggered a "is this worth it?" review,
  because the system has no mechanism that asks. Adding is rewarded; the recurring cost is invisible.
- **Asymmetric retirement.** Adding a gate is a celebrated discovery-step; retiring one requires a
  justified changeset, and when the `independent-review`/`ux-audit-closure` gates *were* retired
  (563), the retirement was **reverted and relitigated** (visible in the commit log). Removal has
  friction; addition does not.
- **The sharpest finding — the reward is for *authoring*, not *efficacy*.** Phase 1 surfaced three
  patterns that this single incentive predicts exactly:
  1. **Speculative bulk-authoring** — `dead-code`, `test-to-code`, `module-deps` shipped in one
     2026-05-21 commit, never re-touched, zero catches. Agents added gates because *adding* is the
     rewarded act, with no demonstrated failure motivating them.
  2. **Dormant wiring** — `ssot-catalog-sync` (guards a *shipped* bug) and `host-owns-truth`/
     `runtime-witness` (soft-fail without a snapshot) were authored but never fully wired. The
     credit lands at creation; ongoing efficacy is nobody's reward.
  3. **Opt-in meta-gates with zero adopters** — `tempdoc-wiring`, `adr-coverage` (a gate-invented
     convention). Infrastructure built for a critical mass that never came.
- **Legibility capture (the root).** Governance is the only work the system *instruments about
  itself* (registries, SARIF, dashboards, changesets) while user-facing value is uninstrumented.
  For an autonomous agent choosing the next task, governance offers a guaranteed green check with no
  live-stack/UX risk. The gradient points at gates regardless of marginal value — and there is no
  counterweight metric pulling the other way.

Two honest readings:

1. **The "governance reduced defects" claim is *unprovable from this data* — because reverts/
   hotfixes were already near-zero before the substrate existed** (0 in Feb–Mar). You cannot
   demonstrate a reduction in a defect class that was already at floor. This is the predicted
   attribution weakness; we do **not** claim governance cut defects. (The per-gate true-positive
   catches in §B.1 — wire ×3, consumer-drift, execution-surface ×2, exception-count — are the
   honest evidence of value, not the aggregate trend.)
2. **A governance-*induced* cost class appeared exactly when the substrate landed.**
   "Green/unbreak the gate" commits were **0 before May, then 12 (May) + 5 (Jun)** — the gates
   themselves became a source of merge-time build breakage requiring rescue. This is real
   self-inflicted churn the machinery generates (especially on cross-worktree merges that inflate
   ratchet baselines). It is the empirical face of the maintenance cost §B.1 flagged.

### §B.5 — Phase 5: synthesis & verdict

**The headline, corrected.** "Agents work too much on governance gates" is **true in its
conclusion but wrong in its target.** The problem is not the *number* of gates — the 35-gate count
is mostly justified, and ~13 of them (the structural/arbitration + surface-coverage families) are
cheap, near-zero-maintenance, and each makes a *real, named AI-agent failure mode* unauthorable
(hand-rolled modal, forked SearchTrace, a second top-level window, a proto back-compat break). For
a codebase authored by parallel agents that don't inherit `CLAUDE.md` and carry no cross-session
memory, that core is *more* justified than it would be in a human repo, not less.

**Where the over-investment actually is** (4 concrete loci):
1. **One ratchet dominates the churn.** `class-size` (101 commits / 64 changesets) is a pin-bump
   treadmill that documents LOC creep without driving decomposition. It is the single largest
   mechanical source of the commit volume the requester noticed.
2. **~5 speculative gates that never paid off** — `dead-code`, `test-to-code` (unwired),
   `module-deps` (redundant with ArchUnit), `tempdoc-wiring` (0 adopters), `adr-coverage`
   (circular). Bulk-authored on incentive, not on evidence of failure.
3. **Self-inflicted rescue churn** — the 17 "green the gate" merge-rescue commits (§B.4) that did
   not exist before the substrate.
4. **The incentive itself** — a one-way ladder that rewards *authoring* gates and is blind to
   *maintenance cost* and *catch rate*, with no value counterweight (§B.3).

**The counter-twist that keeps the critique honest:** the substrate is simultaneously
**over-built and under-wired.** `ssot-catalog-sync` guards a bug that *actually shipped*, yet runs
in no pipeline; `host-owns-truth`/`runtime-witness` soft-fail silently without a snapshot. The
remedy is not a blanket cut — it is *prune the dead weight AND finish the real protection.*

**Recommendations, each with a falsifiable prediction** (analysis-only; execution is a future
tempdoc per §7):
- **R1 — Add a cost side to the ladder.** A gate must justify its rolling maintenance/changeset
  churn against catches in a window; a gate with zero catches AND an inert failure class is a
  retire candidate. *Prediction: retiring `test-to-code`/`dead-code`/`tempdoc-wiring`/`adr-coverage`
  will not increase any real defect rate over the following N weeks.* (Falsified if a defect they'd
  have caught ships.)
- **R2 — Stop using `class-size` as a refactor driver; convert to a hard-cap-only backstop.**
  *Prediction: removing the per-file shrink ratchet (keeping a single hard ceiling) eliminates the
  bulk of its 64 changesets with no increase in god-class incidents.* **The complementary direct
  action — fixing the actual offenders — is chartered in tempdocs 583 (LocalApiServer, 37
  changesets), 584 (AgentLoopService, 15), 585 (AgentController, 7); those three files account for
  ~59 of the 64 class-size changesets. Each charter is remedy-open (decomposition OR rewrite), not a
  decomposition mandate.**
- **R3 — Freeze the meta-tier count.** No net-new gate-governing-gates beyond the proven two
  (`exception-count`, `prose-tier-register`). *Prediction: frozen meta-tier ⇒ no rise in register
  rot.*
- **R4 — Finish wiring before authoring.** A gate that soft-fails without a snapshot, or is wired
  into no pipeline, is not "done." *Prediction: wiring `ssot-catalog-sync` into the pre-merge path
  will, within a quarter, catch ≥1 real catalog divergence (the class already shipped once).*
- **R5 — Instrument a value counterweight.** Until user-facing work has a success signal as legible
  as a green gate, the gradient will keep pulling agents toward governance. This is the root cause;
  R1–R4 are symptom control.

**Substrate-level verdict:** The governance machine is **net-positive and largely justified for an
agent-built codebase**, but it has crossed from *paying down a fixed risk* into *self-generating
maintenance*, driven by an incentive that rewards authoring over efficacy. The fix is not
demolition — it is a **cost-side on the ladder, a value counterweight, and a finish-the-wiring
pass** — roughly 6 retire/downgrade/rewire actions, not 35.

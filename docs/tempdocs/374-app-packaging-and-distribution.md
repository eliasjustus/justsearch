# 374. App Packaging & Distribution

Umbrella tempdoc for getting JustSearch to a ship-ready, downloadable installer.
Subsumes the open gaps from tempdocs 191 (Tauri shell) and 202 (installer maturity)
into a single actionable plan. This document focuses on **what exists today** and
**what remains** before a public release.

## Current State Audit (2026-03-30)

### G16 Root Cause: Bundle CUDA Failure

When the headless backend is launched from the sidecar bundle, ORT CUDA
sessions fail to initialize and enrichment falls back to CPU (~4x slower).
The dev backend via `runHeadlessEval` uses GPU fine. Same code, same models.

**Root cause:** Two env vars set by Gradle are missing from the Tauri sidecar:

| Variable | Gradle `runHeadlessEval` | Tauri sidecar |
|----------|--------------------------|---------------|
| `JUSTSEARCH_ONNXRUNTIME_NATIVE_PATH` | `<repo>/tmp/ort-variant-test/cuda-12.4-v1.24.3` | not set |
| `JUSTSEARCH_GPU_ENABLED` | `true` | not set |
| `justsearch.onnxruntime.native_path` sysprop | set (line 1704 of build.gradle.kts) | not set |

**Failure chain:**
1. `lib.rs:spawn_headless_backend()` sets only `JUSTSEARCH_HOME`. No CUDA paths.
2. `GpuAutoDetection.probe()` looks for `<repoRoot>/tmp/ort-variant-test/cuda-12.4-v1.24.3/`
   but `repoRoot` resolves to `headless_dir` (the bundle), which has no `tmp/`.
3. GPU stays disabled. `OrtCudaHelper.resolveOrtNativePath()` falls back to the
   model directory, which has no CUDA DLLs → `missingDlls` status.
4. The CUDA DLLs (`onnxruntime_providers_cuda.dll`, `cudart64_12.dll`,
   `cublas64_12.dll`, `cublasLt64_12.dll`) are never bundled —
   `bundleSidecarResources` in `build.gradle.kts` doesn't include them.

**Fix requires two things:**
1. **Bundle or download CUDA ORT DLLs** — either include in the sidecar
   (adds ~500 MB), or offer a post-install "GPU Booster Pack" download.
   The llama-server CUDA variant already uses this pattern (`variants/cuda12/`
   staged by `stageLlamaCudaVariant`).
2. **Set `JUSTSEARCH_ONNXRUNTIME_NATIVE_PATH`** in `lib.rs` pointing to
   wherever the DLLs land (e.g., `<app_data_dir>/native-bin/ort-cuda/`),
   using the same version-file-based upgrade pattern as llama-server
   (lines 434–525 of lib.rs).

**Key files:**
- `modules/ui/build.gradle.kts:1637-1677` — Gradle auto-detection logic
- `modules/shell/src-tauri/src/lib.rs:537-578` — Tauri spawn (missing env vars)
- `modules/ort-common/.../GpuAutoDetection.java:52-98` — conventional path search
- `modules/ort-common/.../OrtCudaHelper.java:107-144` — native path resolution
- `modules/ort-common/.../OrtSessionManager.java:368-412` — GPU session creation

### G17 Root Cause: LLM Activation in Bundle

Two independent LLM activation stacks exist:

| Stack | Entry point | Variant check | Model path | Used by |
|-------|-------------|---------------|------------|---------|
| `RuntimeActivationService` | `POST /api/ai/runtime/activate` | Yes — needs `variants/{id}/llama-server.exe` | `UiSettings.getLlmModelPath()` | UI "Start AI" button, production flow |
| `BootstrapInferenceFactory` | `JUSTSEARCH_AI_AUTOSTART_ENABLED=true` | No — uses `InferenceConfig.fromEnvironment()` | `JUSTSEARCH_LLM_MODEL_PATH` or GGUF auto-discovery | `runHeadlessEval -Pllm`, eval/dev flow |

**The bundle backend hits `RuntimeActivationService`** (the production stack),
which checks for `{aiHome}/native-bin/llama-server/variants/{variantId}/llama-server.exe`.
The baseline CPU `llama-server.exe` is copied flat (not under `variants/`), so
there are no installed variants. Even if `"cuda12"` exists in `variants/`,
the activation also requires `UiSettings.getLlmModelPath()` to be non-empty
(reads from `settings.json`), which is empty on a fresh install.

**The dev backend uses `BootstrapInferenceFactory`** (the autoStart stack),
which reads `InferenceConfig.fromEnvironment()` and discovers the server exe
and GGUF model via env vars / filesystem scanning. No variant directory or
settings file needed.

**Workaround for agent testing:** Set autoStart env vars when launching the
bundle backend:

```
JUSTSEARCH_AI_AUTOSTART_ENABLED=true
JUSTSEARCH_LLM_MODEL_PATH=<path-to-gguf>
JUSTSEARCH_MODELS_DIR=<models-dir>
```

This bypasses the variant install system entirely. Valid for eval/testing,
not for production (users should control when AI activates).

**Production fix (369 analysis):** Treat the baseline exe as a virtual
"default" variant. Two changes:

1. **`runActivate()`**: When `variantId` is `"default"` and
   `variants/default/llama-server.exe` doesn't exist, fall back to the
   flat baseline at `{variantsRoot}/../llama-server.exe` (i.e., one
   level up from `variants/`). The self-test still runs against this exe.
   The model path gate (line 343) remains — users must complete "Install AI"
   before "Start AI" works, which is the intended UX.

2. **`listInstalledVariants()`**: When the baseline exe exists flat in
   `{variantsRoot}/../llama-server.exe` but not under `variants/`, add a
   synthetic `InstalledVariant("default", baselinePath)`. This makes it
   visible to the UI so the "Start AI" button has something to activate.

This preserves the self-test safety check, doesn't require bundling
changes, and doesn't unify the two stacks (they serve different
purposes: Path A for eval autostart, Path B for production UI with
user-controlled activation and GPU self-test).

**Edge cases (not real problems):** The self-test returns
"inconclusive" on CPU-only machines (no NVML), blocking Advanced
mode activation. Irrelevant: Simple mode "Start AI" calls
`switchToOnlineMode()` directly, bypassing `RuntimeActivationService`.
`gpuLayers=99` is forced on activation (line 403) — cosmetically
wrong for CPU baseline but functionally harmless.

**Rejected alternatives:**
- **Create `variants/default/` at bundle time** — ceremony that adds
  no value; the baseline exe IS the default runtime.
- **Unify the stacks** — Path B's self-test, rollback, and
  UiSettings persistence are production requirements that Path A
  correctly skips for eval. Merging them couples eval flexibility
  to production safety constraints.

**Key files:**
- `modules/ui/.../RuntimeActivationService.java:337-340` — variant existence check
- `modules/ui/.../RuntimeActivationService.java:343-353` — model path check
- `modules/ui/.../RuntimeActivationService.java:849-871` — listInstalledVariants
- `modules/ui/.../RuntimeActivationService.java:911-938` — resolveVariantsRoot
- `modules/app-services/.../BootstrapInferenceFactory.java:100-121` — autoStart
- `modules/app-services/.../AppFacadeBootstrap.java:849-860` — autoStart flag
- `modules/app-inference/.../InferenceConfig.java:72-173,280-333` — path discovery

### Agent Testing & Debugging Capabilities

Understanding what agents can and cannot verify is critical for planning QA
coverage of the packaged app.

**What agents can do today (API/CLI — no UI):**

| Capability | Tool | Notes |
|------------|------|-------|
| Start/stop backend | `justsearch.dev.start/stop`, jseval `start_backend()` | Full lifecycle management |
| Search queries | `justsearch.dev.search_query`, `POST /api/knowledge/search` | Filters, facets, excerpts, pagination |
| Eval pipeline | `python -m jseval run` | Ingest, enrich, search, nDCG@10 metrics |
| A/B regression | `python -m jseval compare --fail-on-regression` | Per-query rank analysis |
| Hot-reload code | `justsearch.dev.reload` | ~2-3s round-trip, no restart |
| Index reset | `POST /api/debug/reset-index` | Eval mode only |
| Health/status | `/api/health`, `/api/status`, `/api/debug/state` | ~40 REST endpoints total |
| LLM activation | `justsearch.dev.ai_activate` | Polls until online |
| Agent chat (SSE) | `justsearch.dev.agent_chat` | End-to-end agent testing |
| Log tailing | `justsearch.dev.tail_log`, `jseval logs` | Head + Worker logs |
| Bundle smoke test | `./gradlew :modules:ui:smokeSidecarBundle` | Verifies production bundle headlessly |
| Preflight checks | `justsearch.dev.preflight` | GPU, models, stale processes |

This covers backend development, search quality, pipeline profiling, and
regression testing comprehensively.

**What agents cannot do today:**

- See or interact with the **Tauri desktop window** (native, not a browser)
- Verify **visual layout**, component rendering, CSS, or animations
- Test the **search-to-inspect flow** as a user experiences it
- Exercise the **settings panel**, library management, or onboarding via the UI
- Validate the **NSIS installer wizard UX** (screen flow, progress bar, branding)

### Autonomous Install & Test Feasibility (2026-03-30)

An agent can autonomously install and run JustSearch from the built installer
without any GUI interaction.

**NSIS silent install (fully non-interactive):**

NSIS natively supports CLI-only installation — no wizard, no windows, no prompts:

```
JustSearch_*-setup.exe /S /D=D:\JustSearch-install
```

Statistically equivalent (all p > 0.05). Pipeline: bundle 8% faster (253s vs
276s). All models matched (fingerprint, GPU, backend). Ranking alignment
and divergence analysis below.

*Ranking alignment:*

| Level | Rate |
|-------|------|
| Top-1 match | 92.7% (278/300) |
| Top-3 set match | 83.0% |
| Top-10 exact order | 46.3% (139/300) |
| Same docs, different order | 15.3% (46/300) |
| Different candidate set | 38.3% (115/300) |

*Divergence analysis (161 non-identical queries):*

**Source of differences:** 115/161 queries have different candidate sets
entering the cross-encoder (avg 8.8/10 docs shared, 97 queries differ by
exactly 1 doc). This is first-stage retrieval non-determinism (dense kNN +
SPLADE), not CE reranking. The remaining 46 queries have identical
candidate sets but different CE ordering.

**Score gaps at first divergence:** 0 queries have gap >= 1.0. 87 queries
have gap 0.01-0.1 (CE float non-determinism). 49 queries have gap < 0.01
(effective ties). 25 queries have gap 0.1-1.0 (different candidates).

**Top-1 mismatches (22 queries):** Dev's #1 appears in bundle's top-10 for
20/22 cases, at avg rank 3.0, median 2.0. Minor displacement, not missing
documents.

**NER entity count:** dev=5369, bun=6100 (+14%). Same model, same GPU. The
count is docs-with-entities, not ORT calls (both ~12K calls). Likely caused
by batch-ordering non-determinism: different indexing order → different batch
composition → different per-document NER outputs (batch padding affects model
predictions). Entity boost is 0.0 in searchConfig, so this does not affect
ranking.

**Root cause:** Reproducible across runs. The first-stage fusion (BM25 +
dense + SPLADE) combines three non-deterministic signals. Dense kNN uses
approximate nearest neighbors over HNSW, which is sensitive to insertion
order. SPLADE sparse vector scores have floating-point accumulation order
differences. These compound at the top-K boundary, swapping 1-2 borderline
docs. The CE then reranks a slightly different candidate set, producing
different final rankings. **This is inherent non-determinism, not a bug.**
nDCG@10 is statistically identical (p=0.099, 95% CI: [-0.025, +0.002]).

**Conclusion:** Agents can autonomously install, boot, and interact with the
full JustSearch stack including UI and LLM. Lexical and hybrid+CE search
quality is statistically identical between bundle and dev. The environment diff
methodology (compare diagnostics before eval) correctly validated parity and
caught the reranker lazy-init timing difference. LLM and QU work via the
autoStart workaround. The critical requirement for browser-based frontend
testing is dev mode (`-Djustsearch.prod=false`, the default).

### Validation Limitations (2026-03-31)

**All testing was performed on the development machine.** No test has validated
the app on a clean system. Specific limitations:

1. **Tauri shell (JustSearch.exe) — narrowly launched 2026-04-24, still not
   visually verified.** The rebuilt installer's `JustSearch.exe` was launched
   (PID 17884) and `lib.rs:spawn_headless_backend()` successfully started the
   java.exe child (`/api/health` + `/api/knowledge/search` responded as
   expected). This exercises the **backend-spawn** code path, including the
   G16 env-var wiring. What remains unverified: the WebView2 render path,
   IPC bridge, `invoke('session_token')`, and every visual/interactive aspect
   of the shell. A deadlocked or blank window would have produced the same
   HTTP signals observed here. GPU portion of G16 also unexercised because
   no CUDA DLLs are bundled in the CPU-only alpha. Real closure of this
   limitation requires driving the WebView2 window (tauri-driver / WebDriver,
   or human-observed clean-machine run).
2. **GPU acceleration not in installer.** The 742 MB CPU-only alpha excludes
   ORT CUDA DLLs (2 GB) and llama CUDA variant (1.1 GB). A clean machine
   gets CPU-only enrichment. The G16 code fix exists but has nothing to act on
   until either (a) CUDA is bundled (incompatible with NSIS 32-bit — G21),
   (b) CUDA DLLs ship as a post-install downloadable "GPU Booster Pack",
   or (c) the installer moves off NSIS (e.g., MSIX, Inno Setup 64-bit).
3. **Install AI never completed end-to-end.** GGUF download worked (cancelled
   manually). ONNX downloads from GitHub Releases hit curl SSL timeouts
   (exit 35) on files >300 MB. Models were manually copied from dev repo.
4. **Dev repo pollutes all tests.** Model discovery, SSOT resolution, and
   config snapshots all find dev-machine paths via CWD, repoRoot auto-detection,
   or stale snapshots. Required hiding `models/` dir to isolate.
5. **Production activation flow (RuntimeActivationService) untested.** All LLM
   tests used the `JUSTSEARCH_AI_AUTOSTART_ENABLED` eval shortcut.

**Search quality evaluation methodology:** Used SciFact (BEIR dataset, 5183
docs, 300 queries with binary relevance judgments) via `scripts/jseval/`.
Pipeline: materialize corpus → ingest via API → wait for enrichment → run
queries → compute nDCG@10/P@1/R@10. This requires Python, pip dependencies,
and corpus data — not available in the sandbox environment.

### G2 versioning — recommendations (2026-04-24)

See the dedicated §"G2 versioning research (2026-04-24)" section below
for the full investigation and rationale; the short form lives here
for reference.

**Recommended minimum versioning policy for the alpha phase:**

1. **Schema: Semantic Versioning with pre-release identifiers.**
   Format `MAJOR.MINOR.PATCH[-PRERELEASE]`, e.g. `2.0.0-alpha.1`,
   `2.0.0-alpha.2`, … `2.0.0-beta.1`, then `2.0.0`. Matches Tauri v2's
   updater expectations (should G1 ever be revived) and Cargo's native
   understanding, without committing to a release cadence.
2. **Source of truth: `gradle.properties`.** Already the canonical
   version per `sync-version.ps1`. No change needed to wiring.
3. **Per release:** bump `gradle.properties`, run
   `scripts/ci/sync-version.ps1 -RequireReleaseSemver`, commit, tag
   `v2.0.0-alpha.N`, push tag. CI on `workflow_dispatch` (ADR-0026)
   builds + attaches installer to a GitHub Release named after the
   tag. That closes G3 alongside G2.
4. **Snapshot cadence between releases:** immediately after tagging
   `v2.0.0-alpha.N`, bump `gradle.properties` to
   `2.0.0-alpha.N+1-SNAPSHOT`. The `-SNAPSHOT` suffix is preserved for
   day-to-day dev so there's no ambiguity about whether a given build
   is a release or a work-in-progress.
5. **Tag convention:** `v` prefix (e.g. `v2.0.0-alpha.1`). Matches
   Tauri's own tag conventions, git-tag-parse expectations in most
   tooling, and sorts correctly in GitHub's release UI.
6. **No release calendar yet.** Tag when there's something worth
   shipping. Don't commit to a cadence until there's a reason to.

### G2 versioning research (2026-04-24)

**Current state (as of 2026-04-24):**

- `gradle.properties` — canonical version: `2.0.0-SNAPSHOT`.
- `sync-version.ps1` updates three targets from the canonical value:
  `modules/shell/src-tauri/tauri.conf.json`,
  `modules/shell/package.json`,
  `modules/shell/src-tauri/Cargo.toml`.
  It does **not** update `modules/ui-web/package.json` (pinned to `0.0.0`
  by design — ui-web is versioned independently as a dev artifact,
  not shipped outside the Tauri bundle).
- `sync-version.ps1 -RequireReleaseSemver` enforces `^\d+\.\d+\.\d+$` —
  **strict GA only, rejects pre-release identifiers** like
  `2.0.0-alpha.1`. This is too restrictive for pre-GA phases.
- Without `-RequireReleaseSemver`, the script accepts anything matching
  `^\d+\.\d+\.\d+.*$` including snapshots and pre-release suffixes.
- `.github/workflows/build-installer.yml` is `workflow_dispatch`-only;
  uploads a 30-day artifact; **does not create a GitHub Release** or
  attach assets to one. No tag-triggered workflow exists anywhere.
- `git tag` returns: `2.0`, `phase-1.done`, `v2.0.0-SNAPSHOT` — an
  informal history, not a versioning discipline. The
  `v2.0.0-SNAPSHOT` tag is semantically wrong (snapshots are by
  definition untagged work-in-progress).
- No release automation: no release-drafter, release-please,
  semantic-release, tauri-plugin-updater, or similar. Clean slate.

**Upstream constraints (Tauri v2):**

- Tauri v2 itself uses SemVer with the progression alpha → beta → RC → stable.
  Mirroring that progression is idiomatic.
- `tauri-plugin-updater` does SemVer-aware version comparison. Pre-release
  identifiers (`-alpha.N`, `-beta.N`, `-rc.N`) work correctly because
  SemVer 2.0.0 defines their ordering (numeric-first, then lexical;
  `2.0.0-alpha.2 < 2.0.0-alpha.10` per the spec). Custom comparators must
  be careful — plain lexical string sort would put `alpha.10` before
  `alpha.2`. Not a concern today (updater deferred), but a footgun if G1
  is ever revived with a non-default comparator.
- Tauri v2 migration guide states: Cargo.toml + tauri.conf.json +
  package.json must all have the same version. `sync-version.ps1` handles
  this today.
- Installer filename derives from `tauri.conf.json` version:
  `JustSearch_<version>_x64-setup.exe`. With `2.0.0-alpha.1` the name
  becomes `JustSearch_2.0.0-alpha.1_x64-setup.exe` — legal on NTFS, fine
  for NSIS.

**Option comparison:**

| Option | Versioning scheme | Pros | Cons |
|--------|-------------------|------|------|
| A. Strict GA-only | Only ever tag `2.0.0`, `2.0.1`, … | Matches current `-RequireReleaseSemver` gate. Simple. | Cannot distinguish alpha/beta/rc artifacts. Implies GA-quality for every release. Wrong for the private-alpha phase we're actually in. |
| B. SemVer pre-release identifiers | `2.0.0-alpha.1`, `-alpha.2`, `-beta.1`, `-rc.1`, `2.0.0` | Standard, machine-readable, works with Tauri updater. Mirrors Tauri's own progression. | Requires relaxing the `-RequireReleaseSemver` gate regex. |
| C. CalVer (date-based) | `2026.04.1`, `2026.04.2`, … | No major/minor/patch overhead. Rapid iteration friendly. | Less standard in desktop-app ecosystem. Doesn't map to Tauri updater expectations cleanly. Doesn't signal "is this GA." |
| D. Ad-hoc tags | `alpha-2026-04-24`, `demo-5`, … | Zero ceremony. | Not SemVer-comparable. Breaks Cargo / tauri.conf.json versioning expectations. Blocks any future updater path. |

**Recommendation: Option B** (SemVer with pre-release identifiers).
It's the only option that works natively with Cargo, Tauri's config
schema, and the future updater path while accurately reflecting the
pre-GA phase.

**Concrete implementation plan:**

1. **Relax `sync-version.ps1 -RequireReleaseSemver`.** Current regex
   `^\d+\.\d+\.\d+$` rejects pre-release identifiers. Change to the
   standard SemVer 2.0.0 regex that accepts `MAJOR.MINOR.PATCH[-PRERELEASE]`
   but still excludes snapshots:

   ```
   ^\d+\.\d+\.\d+(-[A-Za-z0-9]+(\.[A-Za-z0-9]+)*)?$
   ```

   This accepts `2.0.0`, `2.0.0-alpha.1`, `2.0.0-rc.2`; rejects
   `2.0.0-SNAPSHOT`, `2.0.0-alpha.1-SNAPSHOT`, empty, etc.

   Alternative framing if stricter pre-release discipline is wanted:
   whitelist only `alpha.N` / `beta.N` / `rc.N` suffixes:

   ```
   ^\d+\.\d+\.\d+(-(alpha|beta|rc)\.\d+)?$
   ```

   Recommend the first (looser) variant — fewer surprises when
   project conventions evolve.

2. **Establish the dev/release convention explicitly in a CONVENTIONS
   doc or in `gradle.properties` comment**:
   - Working copy of `gradle.properties` always has a `-SNAPSHOT`
     suffix during development.
   - Release process strips `-SNAPSHOT`, runs
     `sync-version.ps1 -RequireReleaseSemver` (which now accepts
     pre-release identifiers), commits, tags `v<version>`, pushes.
   - Immediately after tagging, bumps `gradle.properties` to the next
     working version + `-SNAPSHOT`.

   Example walk-through starting from today:
   | Step | gradle.properties | Action |
   |------|-------------------|--------|
   | now | `2.0.0-SNAPSHOT` | — |

   | post-release | `2.0.0-alpha.2-SNAPSHOT` | commit, continue dev |
   | prep release 2 | `2.0.0-alpha.2` | sync-version, commit, tag `v2.0.0-alpha.2`, build + publish |
   | … | … | … |
   | prep beta | `2.0.0-beta.1` | sync-version, commit, tag `v2.0.0-beta.1`, … |
   | prep GA | `2.0.0` | sync-version, commit, tag `v2.0.0`, … |
   | post-GA | `2.0.1-SNAPSHOT` | continue dev |

3. **Tag convention: `v` prefix.** Matches Tauri's own repo

#### New structural findings

##### Finding #4 — `GpuAutoDetection` is a dev-tree filesystem check, not a CUDA probe (CRITICAL)

`io.justsearch.ort.GpuAutoDetection` looks for `onnxruntime_providers_cuda.dll` at:
1. `ORT_NATIVE_PATH` / `-Djustsearch.onnxruntime.native_path` (env override, unset in production)
2. `<repo-root>/tmp/ort-variant-test/cuda-12.4-v1.24.3/` (a dev-tree path that doesn't exist in production installs)

If neither contains the sentinel DLL, logs `"GPU auto-detection: no CUDA DLLs found"` and reports `cudaAvailable: false`. **The probe never calls `cuInit`, `cuDeviceGetCount`, or any CUDA API.** Every production install (sandbox, normal user, enterprise, VDI) gets `cudaAvailable: false` regardless of actual hardware. Empirically demonstrated:

- `cuInit(0)` via System32 `nvcuda.dll` → SUCCESS, driver 13020
- `cudaGetDeviceCount` via bundled `cudart64_12.dll` → SUCCESS, 1 device
- `llama-server.exe -ngl 99` with bundled `cuda12` stack → 78.6 tok/s on Llama-3.1-8B (GPU-resident), 67 tok/s on Qwen3.5-9B + mmproj (multimodal, 9.17 GB resident)
- JustSearch's probe during all of the above: `cudaAvailable: false`

`gpu-test-evidence/probe-analysis.md` includes a 50-line drop-in replacement using Java FFM + `nvcuda.dll`'s driver API. Same pattern PyTorch / TensorFlow / ggml-cuda use.

Impact: the chat package is silently skipped for every production user. My Bug 2 fix (Qwen3.5-9B manifest) sits behind a gate that's stuck closed for everyone.

##### Finding #10 — Embedding manifest has no CPU variant (CRITICAL — actual show-stopper)

`embedding` package has only one variant:
```json
{"filename":"model_fp16.onnx","precision":"FP16","targetEP":"CUDA"}
```

Every other package (`splade`, `reranker`, `ner`, `citation-scorer`) ships a CPU+CUDA pair. On a CPU-profile install (the actual non-CUDA path), the FP16/CUDA file downloads anyway; the Worker rejects it as `degraded: true` and never instantiates `OrtSession`. Documents accumulate in `embeddingQueueSize` indefinitely. Hybrid retrieval is silently broken.

The `degradationReason` text — `"FP16 model on CPU (significant performance overhead — re-run Install AI to get CPU variant)"` — is a lie: there is no CPU variant for Install AI to fetch.

##### Finding #11 — Install AI omits `model_manifest.json` from embedding `supportingFiles[]`

The 135-byte file required by `OnnxModelDiscovery`:
```json
{"cpu":"model_fp16.onnx","gpu":"model_fp16.onnx","tokenizer":"tokenizer.json","pooling_config":"pooling_config.json"}
```

Without it: `"ONNX model 'gte-multilingual-base': not found at any standard location"` even with `model_fp16.onnx` on disk. Empirically proven by the agent: copying the file in flips discovery on the next boot.

Currently masked by Finding #10 (FP16 rejection happens after discovery), but independent.

##### Finding #11.5 — Head-side `/api/ai/runtime/status` reports `not_found` regardless of disk state

Install AI doesn't write the `justsearch.<feature>.model_path` config keys. Head's `resolveOnnxFeatures()` reads those and reports `reason: "not_found"` whenever they're `<unset>`. Independent of #10/#11; persists across boots.

##### Finding #4.5 — `installedVariants[].exePath` lies about file existence

`/api/ai/runtime/status` reports `variants/cuda12/llama-server.exe`. That file doesn't exist — `cuda12/` ships only the CUDA-specific DLLs (cudart, cublas, cublasLt, ggml-cuda) plus parallel ggml-cpu backends. Runtime activation must merge files from `default/` and `cuda12/` somewhere; the published `exePath` doesn't reflect the actual launch location.

#### What this indicates structurally

Three patterns repeat across these findings:

1. **"Install AI completed" doesn't mean "AI works."** Three independent silent-failure paths: chat skipped on probe-says-no-CUDA (broken probe), embedding silently rejected on EP mismatch, head-side config keys never written. The `state: "completed", message: "AI installed."` envelope needs to be a guarantee, not a hopeful summary. (This is a stronger version of the deferred Bug 3c.)

2. **Probes that work in dev, fail in production.** `GpuAutoDetection` checks a dev-tree path. `OnnxModelDiscovery` requires a file the dev models dir happens to have but Install AI doesn't ship. `JUSTSEARCH_MODELS_DIR` half-works. Each is "we tested in dev where it worked, we shipped." The missing `smokeInstalledBundle` test infrastructure (flagged in the prior session's verification gap) would catch the entire class.

3. **My alpha.8 fixes were too narrow — addressed reported symptoms without verifying the gate.** Bug 3b improved the skip-reason text but didn't ask "is the probe actually correct?" Bug 4 added per-source labels but didn't surface the actual gating probe. Reading `GpuAutoDetection` source on the prior round would have found Finding #4 directly.

#### Lessons learned

1. **"GPU detection works" verified at one consumer is not "GPU is used" verified end-to-end.** Eight `cudaAvailable` consumers were enumerated in the alpha.10 tempdoc; only one (the install gate) was migrated. The others kept calling legacy probes.
2. **Multiple legacy CUDA detection paths must each be migrated explicitly.** There is no central place to fix. `InferenceConfig.detectCudaAvailable`, `HeadlessApp.maybeAutoSelectCuda12Variant`'s `ConfigStore.globalOrNull()` race, and `OnlineAiServiceImpl.applyOverrides`'s `>= 0` precedence are independent defects from independent eras of the codebase.
3. **`UiSettings.gpuLayers = 0` (the default) was a silent override** — it overwrote the resolved config every time Install AI completed. The code's "0 means CPU" semantic was implicit; it should mean "unset, defer to resolved config".

#### Alpha.13 follow-up — Phases E + F (post-implementation critical-analysis pass)

After A1/A2/A3/B/P2 landed (commit `b8ed42a8e`) I ran a critical-analysis pass per CLAUDE.md and surfaced two structural defects that rendered alpha.13 effectively a no-op for the **default install** case:

1. **A1 used `cudaAvailable = layers > 0`** as the proxy. But on a default install (no env var, no `settings.json`, no UI activate), `rc.ai().gpuLayers()` is **0** because `GpuAutoDetection.probe` never writes `gpu_layers` — only `gpu.enabled` and `onnxruntime.native_path`. There is no VRAM-tier auto-population code anywhere in the boot path. Net: A1 helps the env-var-set case but not the default install case (which is what most users hit).
2. **`ConfigStoreRebuilder.rebuild` does not call `contributeAutoDetected`.** Auto-detected values live only in the transient ord-150 map. After every Install AI completion (or my new `applyOrtNativePath`) calls `rebuild`, the master `justsearch.gpu.enabled` flips from true → false because the only source of "true" was lost. The per-feature `gpuEnabled` (computed as `master AND policyAllowed AND perFeature`) flips false. ORT GPU sessions stop being created in the worker, defeating Bug B end-to-end.

Both gaps were missed in my pre-implementation plan because I echoed an earlier (incorrect) tempdoc claim that "ord-150 auto-detect already integrates `gpu_layers`" without verifying with a grep. A 5-minute search of `Map.put` calls inside `GpuAutoDetection.probe` would have caught it.

##### Phase E — sysprop-mirror auto-detected values

`HeadlessApp.augmentGpuAutoDetectionAndMirror` (new, called from `main` right after `GpuAutoDetection.probe`) iterates the probe result map and, for each entry whose corresponding `EnvRegistry.<key>.get()` is empty (no user override at sysprop or env-var level), calls `SystemPropertyUtils.setSysPropIfBlank(key, value)`.

Effects:

- Auto-detected `justsearch.gpu.enabled` and `justsearch.onnxruntime.native_path` now persist through `ConfigStoreRebuilder.rebuild` (which reads sysprops at ord 500 via `contributeEnvRegistry`).
- They also propagate to the worker subprocess via `WorkerSpawner.WORKER_FORWARDED_PROPS` — `GPU_ENABLED` (line 70) and `ORT_NATIVE_PATH` (line 79) are both in that list, forwarded as `-D` JVM args.
- A user's explicit `JUSTSEARCH_GPU_ENABLED=false` is preserved (the EnvRegistry check returns the env value before the mirror runs).

##### Phase F — VRAM-tier auto-population for `gpu_layers`

In the same helper, after the Phase E mirror, compute `shouldUseGpu = userOverride.isPresent() ? Boolean.parseBoolean(userOverride.get()) : autoDetectedSaysTrue`. If `shouldUseGpu` AND no explicit `gpu.layers` / `llm.gpu_layers` is set, query `NvmlService.probe()` for total VRAM. Threshold matches `HardwareProfile.MINIMUM_VRAM_FOR_GGUF` (7.5 GB):

- `vramBytes >= 7.5 GB` → set both `justsearch.gpu.layers=99` AND `justsearch.llm.gpu_layers=99` (different consumers read different keys; both are needed). Add `gpu.layers=99` to the augmented map so `contributeAutoDetected` propagates it at ord 150 too.
- Below threshold → leave at 0. Qwen3.5-9B Q4_K_M (~5.5 GB plus context) wouldn't fit safely; partial offload is an OOM hazard. User can still force layers via env var.

##### Phase I — Regression test (per CLAUDE.md "Audit-driven fixes need a runnable test")

`modules/ui/src/test/java/io/justsearch/ui/HeadlessAppGpuAutoPopulateTest.java` — 7 cases pin the boundary behavior:

1. `autoDetectedGpu_with12gbVram_populatesGpuLayersToFullOffload` — happy path
2. `userExplicitGpuLayersZero_isPreserved` — idempotency / explicit override
3. `userExplicitGpuDisabled_skipsAutoPopulateEvenWhenProbeSaysAvailable` — wrong-gate guard (user `false` wins over autoDetected `true`)
4. `belowVramThreshold_skipsAutoPopulate` — threshold guard
5. `thresholdMatchesHardwareProfileMinimumVramForGguf` — pins the threshold to the canonical constant

The test stubs the `LongSupplier` for VRAM bytes, so it doesn't depend on the host actually having NVML. Pure unit-test discipline. Without this test, an off-by-one mistake in the precedence direction (user `false` flipped to autoDetected `true`, or the threshold comparison reversed) would fail at sandbox round 5; with the test, it fails in milliseconds during the next compile.

#### Bug D root cause — found

After committing the alpha.16 D3 defensive backstop, I refused to accept the deferral and dug for the real root cause. **`WORKER_FORWARDED_PROPS` in `WorkerSpawner.java` was missing the per-encoder GPU flags.** Master `GPU_ENABLED` and global `GPU_LAYERS` were forwarded, but `EMBED_GPU_ENABLED`, `SPLADE_GPU_ENABLED`, `NER_GPU_ENABLED` were not. So sysprops written at the head process (by my D3 boot-time mirror, by Install AI's `applyOnnxSettings`, by anything) never propagated to the worker subprocess via `-D` args. The worker only ever saw whatever was in the snapshot file (written once at head boot) plus its own env vars.

Why my unit tests passed: they exercised the resolution chain in-process. The bug was purely in the cross-process sysprop forwarding. Why the agent's env-var workaround worked: env vars ARE inherited by subprocesses on Windows; the JUSTSEARCH_*_GPU_ENABLED env vars reached the worker through the OS, not through WORKER_FORWARDED_PROPS.

Fix: added `EMBED_GPU_ENABLED`, `SPLADE_GPU_ENABLED`, `NER_GPU_ENABLED` to `WORKER_FORWARDED_PROPS`. Combined with the D3 boot-time mirror, the sysprop chain at the head now has values, and they forward to the worker via `-D` at ordinal 500.

This kills two birds: it fixes Bug D's immediate symptom (per-encoder GPU enabled at the worker) AND fixes a latent gap that any future per-encoder sysprop write at the head would have hit (e.g., `applyOnnxSettings`'s sysprop writes for the model paths reached the worker only via the `EMBED_ONNX_MODEL_PATH` etc. that ARE in WORKER_FORWARDED_PROPS — a coincidence that masked the broader pattern).

Lessons learned:
- "Defer root-cause investigation" is a valid tactical call when the defensive backstop genuinely solves the problem. But if the defensive backstop relies on sysprop propagation (as D3 did via my expectation that head sysprops reach worker), and the propagation itself is broken, the backstop doesn't help. Always verify the propagation channel.
- Static read of resolution logic + unit test ≠ runtime correctness across processes. Cross-process state requires at least one integration test that exercises the actual worker subprocess.
- The user's "no point starting another sandbox round when there's a known bug" pushback was correct. The D3 backstop alone wouldn't have unblocked ONNX GPU on round 7 — and the symptom would have looked identical to the round-6 evidence.

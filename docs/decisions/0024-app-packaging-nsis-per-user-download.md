# ADR-0024: App Packaging: NSIS, Per-User Install, Download-on-Demand

## Status
Accepted

> **Update (2026-07-02, tempdoc 657) — what "download-on-demand" covers today.**
> Decision #2 below ("Models downloaded post-install") has since narrowed. The NSIS
> bundle now **includes** the ONNX search-runtime models (embedding, SPLADE, reranker,
> NER, citation ≈ 3.5 GB) plus the CPU llama-server, staged by `stageOnnxModels`
> (`includeOnnxModels` defaults to `true`, `modules/ui/build.gradle.kts`). So a fresh
> install does full hybrid neural **retrieval** offline with no post-install download.
> Only the **GGUF chat pack** (chat ≈ 5.5 GB + vision projector ≈ 918 MB) and the
> **CUDA runtime** DLLs remain download-on-demand via "Install AI" (VRAM-gated). The
> "~748 MB installer / no models bundled / all ~8.5 GB post-install" figures below
> describe the original design, not the current build. Tempdoc 657 adds an
> install-intent axis and a per-tier weight breakdown so this retrieval-vs-LLM split is
> explicit to users.

## Context

JustSearch needs a Windows desktop installer. Three interlocking constraints shape the packaging decisions:

1. **Model payload size**: The total model set is ~8.5 GB (ONNX embedding, reranker, citation, NER, SPLADE models + GGUF chat model). This exceeds the NSIS 32-bit PE limit (~2 GB) by a wide margin — models cannot be bundled in the installer.
2. **Enterprise compatibility**: Many enterprise Windows configurations restrict MSI installations or require admin elevation. JustSearch is a personal productivity tool that should install without IT intervention.
3. **Cold-start performance**: The JVM backend has a ~6s cold start. JEP 514 AOT caches can reduce this, but they must be generated at build time and bundled.

The installer must package: a jlink-stripped JDK 25 runtime, the Tauri desktop shell, Head and Worker backend JARs (~164 JARs), llama-server binary, SSOT catalogs, AOT caches, and configuration. The lean total (without models) is ~748 MB.

## Decision

Three interlocking packaging decisions:

### 1. NSIS over MSI/WiX
Use NSIS (Nullsoft Scriptable Install System) as the installer format. NSIS produces a single `.exe` installer that does not require admin elevation. Installation target is per-user at `%LOCALAPPDATA%\JustSearch`.

NSIS hooks handle process lifecycle:
- Pre-install kills running JustSearch processes (15s timeout).
- Pre-uninstall does the same.
- Creates `%COMMONAPPDATA%\JustSearch` for enterprise policy files.
- Desktop integration: tray icon, close-to-tray, autostart with `--minimized`, single instance.

### 2. Models downloaded post-install
Models are NOT bundled in the installer. They are downloaded post-install via the "Install AI" flow in the application UI. The lean installer (~748 MB) stays well under the NSIS 32-bit PE limit. Users download ~8.5 GB of models on first run through a managed download pipeline with progress tracking.

### 3. AOT caches bundled at build time
JEP 514 two-step AOT caches for Head and Worker are generated by `generateHeadAotCache` / `generateWorkerAotCache` Gradle tasks and bundled in the installer. These reduce cold-start time by pre-compiling commonly used code paths.

The build pipeline is: `npm run tauri build --bundles nsis` triggers `beforeBundleCommand` which runs `:modules:ui:bundleSidecar`, assembling all components into the Tauri sidecar resource bundle. Code signing is wired but optional (reads PFX from env vars, skips gracefully if not configured). Version sync (`sync-version.ps1`) propagates `gradle.properties` version to `tauri.conf.json`, `package.json`, and `Cargo.toml`.

## Consequences

**Positive:**
- Installer under 2 GB NSIS limit at ~748 MB — fast download and install (~30s).
- No admin elevation required — works in enterprise environments with restricted privileges.
- Per-user install to `%LOCALAPPDATA%` avoids Program Files permission issues.
- AOT caches reduce cold-start time for the JVM backend.
- Clean separation: installer handles app binaries, "Install AI" handles model assets.

**Negative:**
- First-run requires ~8.5 GB model download — poor offline/air-gapped experience.
- "Install AI" UX adds a separate step after installation before the app is fully functional.
- The download pipeline needs resilience (abort-on-first-failure bug exists as of tempdoc 374).
- Users must have internet access after installation for full functionality.
- Windows-only: hard-locked to `x86_64-pc-windows-msvc` with no macOS/Linux targets.

## Alternatives Considered

### MSI/WiX installer
Use the standard Windows Installer (MSI) format with WiX toolset. Rejected because MSI installations frequently require admin elevation on enterprise configurations. The WiX toolset is also more complex to script than NSIS for custom lifecycle hooks (process killing, tray integration). Per-user MSI installations exist but are less reliable across Windows versions than per-user NSIS.

### Bundle models in the installer
Include all ONNX and GGUF models directly in the installer for an offline-capable experience. Rejected because the combined payload (~8.5 GB sidecar + ~748 MB app = ~9.2 GB) exceeds the NSIS 32-bit PE limit. NSIS is a 32-bit executable format and cannot handle installers larger than ~2 GB.

### Squirrel / electron-builder
Use Squirrel (auto-updating installer framework) or electron-builder's installer infrastructure. Rejected because neither is compatible with Tauri's sidecar architecture. JustSearch uses Tauri (Rust + WebView2) with Java sidecar processes — Squirrel and electron-builder assume Electron's process model and would require significant adaptation.

### MSIX (Microsoft Store)
Use MSIX packaging for distribution through the Microsoft Store or sideloading. Rejected because MSIX requires either Store submission (with review process and restrictions) or enterprise sideloading certificates. Code signing for MSIX is more restrictive than NSIS (requires EV certificate for Store, or organization certificate for sideloading). Too much friction for an indie project at this stage.

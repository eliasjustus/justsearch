# Agent Prompt Surface Governance

This document defines where agent-facing instructions and prompt-like text belong.
It is about prompt ownership and drift control, not measuring agent quality.

## Core Rule

Keep volatile implementation facts out of broad always-loaded prompt surfaces.
Route agents to the smallest current source of truth that can answer the task.

Use this ownership order:

1. Current source code and generated contracts own exact implementation facts.
2. Canonical docs under `docs/explanation/`, `docs/reference/`, `docs/how-to/`, and `docs/decisions/` explain those facts.
3. Generated skills mirror canonical docs.
4. Hand-authored skills and rules describe workflows and routing.
5. `CLAUDE.md` stays a compact router and invariant list.

## Surface Inventory

| Surface | Owner | Edit Mode | Notes |
| --- | --- | --- | --- |
| `CLAUDE.md` | repo-agent instruction owner | hand-authored | Keep stable invariants and skill/doc routing here. Do not duplicate long workflow details. |
| `.claude/skills/*/SKILL.md` | skill owner | mixed | Text outside generated markers is hand-authored. Text between generated markers is regenerated by `scripts/docs/skills-sync.mjs`. |
| `.claude/rules/*.md` | rule owner | hand-authored | Keep narrow operational lessons and workflow rules here. |
| `.claude/settings*.json` | hook/rule wiring owner | hand-authored | Treat hook mappings as prompt routing because they decide which instructions appear during work. |
| `scripts/agent-analytics/hooks/*.mjs` | hook owner | code-owned prompt surface | Hook messages and injected guidance affect agent behavior and must be reviewed like prompts. |
| `docs/llms.txt` | docs index owner | generated section plus hand-authored header/footer | Regenerate after canonical doc changes. |
| canonical docs | docs owner | hand-authored | Source of truth for architecture, contracts, workflows, and known issues. |
| dev/prod MCP tool schemas | MCP server owner | code-owned | Exact tool names, arguments, and schemas come from implementation, not prose. |
| product runtime prompts | product prompt owner | code/SSOT-owned | `PromptContributor`, `PromptCatalog`, `modules/prompt-support`, and `SSOT/prompts` are user-facing runtime prompts, not repo-agent instructions. |
| CI/eval prompts | CI/eval owner | hand-authored or script-owned | Example: `scripts/ci/url-probe-system-prompt.md`. Keep separate from repo-agent guidance. |
| sandbox prompts | sandbox owner | hand-authored | `scripts/sandbox/*` can drift from `CLAUDE.md`; review when main invariants change. |

## Placement Rules

Put exact implementation facts in the narrowest durable place:

- Exact MCP tool names and argument shapes belong in MCP server code and the canonical MCP reference docs.
- Exact module ownership belongs in canonical architecture docs and module READMEs.
- Exact API fields and endpoint behavior belong in API contracts, controller/route code, and generated contract maps.
- Repeated workflow procedures belong in a skill or rule, not in `CLAUDE.md`.
- Long experiment registers belong in generated skills with narrow triggers.
- Runtime product prompt behavior belongs in `PromptContributor` classes, `PromptCatalog` primitives, `modules/prompt-support`, and `SSOT/prompts`.

When a broad surface needs a volatile fact, link or route to the owner instead
of copying the details.

## Generated Skills

Generated skills are delivery surfaces, not sources of truth.

Do not edit text between:

```text
<!-- generated:start -->
<!-- generated:end -->
```

Update the canonical source doc, then run:

```text
node scripts/docs/skills-sync.mjs
node scripts/docs/skills-sync.mjs --check
```

Large generated skills are acceptable when they carry registers or domain
history, but their triggers must be narrow. For example, `/inference-runtime`
and `/search-quality` should load for deep runtime or retrieval-quality work,
not for ordinary architecture orientation.

## Product Runtime Prompts

Product runtime prompts are separate from repo-agent prompts.

Current runtime prompt surfaces include:

- `PromptContributor` implementations such as RAG QA, summarization, and URL emission grammar contributors.
- `PromptCatalog`, `Prompt`, and `PromptRef` registry primitives.
- `modules/prompt-support` template loading.
- `SSOT/prompts/**` source templates and packaged copies under shell resources.

Do not update these through `CLAUDE.md` or repo-agent skills. Verify runtime
prompt claims against current code and SSOT packaging behavior.

## Review Checklist

Before changing agent prompt surfaces:

- Identify whether the surface is hand-authored, generated, hook-injected, runtime, CI/eval, or sandbox-only.
- Verify exact tool names, module names, endpoints, and class paths against current code.
- Prefer shortening or routing broad surfaces over expanding them.
- If a generated skill is wrong, update its canonical source and regenerate.
- If a large skill is necessary, make the trigger narrow enough to avoid accidental loading.
- If a stale reference is historical, mark it historical or retired instead of replacing it with a guessed current owner.

Useful checks:

```text
node scripts/docs/prompt-surface-inventory.mjs
node scripts/docs/skills-sync.mjs --check
node scripts/docs/llmstxt-generate.mjs --check
node scripts/docs/verify-canonical-doc-links.mjs
```

`verify-canonical-doc-links` currently has known baseline failures. Preserve
or improve the failure list unless the task is specifically to fix canonical
link debt.
